## 二、前端架构

### 2.1 技术栈详解

#### Vue3 + 组合式API

```typescript
// 组合式API示例：useSocket composable
export function useSocket() {
  const socket = ref<Socket | null>(null)
  const isConnected = ref(false)

  function connect() {
    socket.value = io(SOCKET_URL, {
      auth: { token: useAuthStore().accessToken },
      transports: ['websocket'],
      reconnection: true,
      reconnectionDelay: 1000,
    })
    socket.value.on('connect', () => { isConnected.value = true })
    socket.value.on('disconnect', () => { isConnected.value = false })
  }

  return { socket, isConnected, connect }
}
```

**选型理由**：
- 组合式API比选项式API更适合逻辑复用和TypeScript
- `ref`/`computed`/`watch` 响应式系统直观高效
- `<script setup>` 语法糖简化代码

#### Vite5

**选型理由**：
- 开发服务器启动 < 300ms（对比Webpack 5-10s）
- HMR热更新 < 50ms
- 原生ESM无需打包，开发体验极佳
- Rollup生产构建，代码分割优秀

#### Pinia

**选型理由**：
- 替代Vuex，API更简洁
- 完整TypeScript推断，无需手写类型
- 支持插件（持久化、DevTools）
- 模块化设计，按需加载

## 三、后端架构

### 3.1 技术栈详解

#### Express + TypeScript

**选型理由**：
- Express 4.x 生态最成熟，中间件丰富
- TypeScript 提供类型安全
- 简洁够用，不过度设计（对比NestJS的装饰器开销）

#### Prisma ORM

**选型理由**：
- Schema-first，数据库模型即代码
- 自动生成TypeScript类型
- 迁移管理方便（`prisma migrate`）
- 查询API直观，无需手写SQL

#### Socket.IO

**选型理由**：
- 自动降级（WebSocket → HTTP长轮询）
- 房间机制天然适合群聊
- Redis Adapter支持多实例
- 命名空间隔离不同业务

## 四、鉴权架构

### 4.1 双令牌机制

```
登录成功
    │
    ├── Access Token (15min) ──▶ 短期令牌，放行API
    │
    └── Refresh Token (7d) ───▶ 长期令牌，续签Access

请求流程：
  1. 客户端携带 Access Token 请求API
  2. Token有效 → 正常响应
  3. Token过期 → 返回 401
  4. 客户端用 Refresh Token 请求 /auth/refresh
  5. Refresh有效 → 签发新 Access Token
  6. Refresh过期 → 跳转登录页

WebSocket鉴权：
  1. 连接时在 auth 字段携带 Access Token
  2. Socket中间件验证Token
  3. 验证通过 → 加入用户个人房间
  4. 验证失败 → 断开连接
```

## 五、WebSocket架构

### 5.1 命名空间设计

```
/ (默认命名空间)
  ├── 私聊消息
  ├── 群聊消息
  ├── 输入状态
  └── 在线状态

房间机制：
  user:{userId}        ← 每个用户的个人房间（私聊+通知）
  group:{groupId}      ← 群聊房间
```

### 5.2 消息流转

```
用户A发送私聊消息给用户B：

1. A客户端 → message:send {to: B_id, content: "hello"}
2. 服务端 → 验证 + 持久化
3. 服务端 → message:new → user:B_id房间（推送给B）
4. 服务端 → message:ack → A客户端（确认已存储）
5. B阅读 → message:read → 服务端
6. 服务端 → message:read-ack → A客户端
```

## 六、安全设计

### 6.1 安全清单

| 层面 | 措施 | 实现 |
|------|------|------|
| 传输 | HTTPS/WSS | Nginx SSL + 强制HTTPS |
| 存储 | 密码加密 | bcrypt, saltRounds=12 |
| 鉴权 | JWT双令牌 | Access(15min) + Refresh(7d) |
| XSS | 内容转义 | 消息内容HTML转义 |
| CSRF | Token验证 | SameSite Cookie + 自定义Header |
| 注入 | 参数化查询 | Prisma默认参数化 |
| 限流 | 频率限制 | express-rate-limit |
| 上传 | 文件校验 | 类型白名单 + 大小限制 |
| CORS | 跨域控制 | 白名单Origin |
