# 13-账号绑定与验证码设计文档

## 1. 功能概述

用户可绑定邮箱和手机号，绑定流程需通过验证码验证。同时支持验证码登录（无需密码）。

## 2. 主流方案对比

| 软件 | 邮箱绑定 | 手机绑定 | 验证码方式 | 登录方式 |
|------|----------|----------|-----------|----------|
| **Discord** | ✅ 邮箱验证 | ❌ 不支持 | 邮箱验证链接 | 邮箱+密码 |
| **Telegram** | ❌ 不绑定 | ✅ 短信验证码 | SMS OTP | 手机号+验证码 |
| **QQ** | ✅ 可绑定 | ✅ 可绑定 | 短信验证码 | QQ号+密码 |
| **WeChat** | ✅ 可绑定 | ✅ 已绑定 | 短信验证码 | 手机号+验证码 |

**MineChat选型**：双通道绑定（邮箱+手机），邮箱验证码优先（开发成本最低），手机号作为扩展。

## 3. 验证码技术选型

### 3.1 邮箱验证码

**方案**：Node.js 内置 `nodemailer` + SMTP 服务

| 项目 | 说明 |
|------|------|
| 发送库 | `nodemailer` (成熟稳定，npm周下载3M+) |
| SMTP服务 | 开发阶段用QQ邮箱/Gmail SMTP，生产用SendGrid/阿里云邮件 |
| 验证码存储 | Redis（key: `verify:{type}:{target}`, TTL: 5分钟） |
| 验证码格式 | 6位数字（随机，前导零补齐） |
| 频率限制 | 同一目标60秒内只能发1次，每天最多10次 |

**为什么不选邮件验证链接？**
- 开发阶段需要公网回调地址（ngrok等），增加复杂度
- 验证码方式体验更快（用户留在App内，不需要切换邮箱）
- Telegram/微信都是验证码，用户习惯

### 3.2 手机验证码（未来扩展）

**方案**：阿里云SMS / 腾讯云SMS

当前阶段仅预留数据库字段和API接口，不实际发送短信（成本高，需要企业资质）。
手机号绑定流程使用模拟验证码（开发阶段固定验证码 `888888`）。

## 4. 数据模型变更

### 4.1 User 表扩展

```prisma
model User {
  // 已有字段...
  phone       String?   @db.VarChar(20)    // 手机号(可选,唯一)
  phoneVerified Boolean  @default(false)    // 手机是否已验证
  emailVerified Boolean  @default(false)    // 邮箱是否已验证(当前email字段已存在)
}
```

### 4.2 Redis 验证码 Key 设计

```
verify:email:{email}        →  "123456"     TTL: 300s (5分钟)
verify:phone:{phone}        →  "123456"     TTL: 300s
verify:rate:{type}:{target} →  发送次数计数  TTL: 86400s (1天)
```

## 5. API 设计

### 5.1 发送验证码

```
POST /api/v1/auth/send-code
Body: { type: 'email' | 'phone', target: string }
Response: { code: 200, message: '验证码已发送' }
```

**业务逻辑**：
1. 校验 target 格式（邮箱格式 / 手机号格式）
2. 检查频率限制（60秒1次，每天10次）
3. 生成6位随机验证码
4. 存入 Redis（TTL 5分钟）
5. 发送邮件/短信
6. 返回成功

### 5.2 绑定邮箱/手机

```
POST /api/v1/user/bind
Body: { type: 'email' | 'phone', target: string, code: string }
Response: { code: 200, message: '绑定成功', data: { email, emailVerified } }
```

**业务逻辑**：
1. 校验验证码（从Redis取出来比对）
2. 检查 target 是否已被其他用户绑定
3. 更新 User 表对应字段 + verified 标记
4. 删除已使用的验证码
5. 返回更新后的用户信息

### 5.3 解绑邮箱/手机

```
POST /api/v1/user/unbind
Body: { type: 'email' | 'phone', code: string }
Response: { code: 200, message: '解绑成功' }
```

**业务逻辑**：
1. 发送验证码到已绑定的邮箱/手机
2. 校验验证码
3. 清空对应字段 + verified=false
4. 至少保留一种绑定方式（不能全部解绑）

### 5.4 验证码登录（扩展）

```
POST /api/v1/auth/login-code
Body: { type: 'email' | 'phone', target: string, code: string }
Response: { code: 200, data: { accessToken, refreshToken, user } }
```

## 7. 安全策略

- **验证码长度**：6位数字，10^6 = 100万种可能
- **验证码有效期**：5分钟
- **频率限制**：
  - 同一目标 60秒/次
  - 同一目标 10次/天
  - 同一IP 50次/天
- **错误尝试**：同一验证码最多验证5次，超过后作废
- **验证码使用后立即删除**：防止重放攻击

## 8. 开发阶段Mock策略

- 邮箱验证码：通过 nodemailer 实际发送（配置QQ邮箱SMTP）
- 手机验证码：固定验证码 `888888`，控制台打印实际验证码
- 频率限制：开发阶段放宽到10秒/次，方便调试
