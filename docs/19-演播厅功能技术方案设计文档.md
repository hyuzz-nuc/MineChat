# 19 - 演播厅功能技术方案设计文档

> 编写日期：2026-07-01 | 版本：v1.0

---

## 一、功能概述

"演播厅"是MineChat的多人同步观影功能模块，支持创建房间、邀请好友、同步播放视频、实时聊天/弹幕。类似SyncTube、Watch2Gether、微光等产品的核心体验。

---

## 二、媒体资源获取方案对比

### 方案A：本地文件上传 + 服务端中转

**原理**：房主上传视频文件到服务端，服务端通过HTTP Range请求分发流媒体。

| 维度 | 评估 |
|------|------|
| 跨域 | 无跨域问题，同源分发 |
| 版权 | 用户自行上传，平台无版权风险 |
| 带宽 | **服务端带宽压力大**，N人同时观看需N倍流量 |
| 存储 | 需大量存储空间，需定期清理 |
| 复杂度 | 中等，需实现分片上传+流式分发 |
| 延迟 | 中等，依赖服务端带宽 |

**适用场景**：小规模（2-5人）私密观影，短时长视频

### 方案B：WebTorrent / P2P去中心化分发

**原理**：房主通过WebTorrent将视频种子的magnet链接分享给房间成员，浏览器之间直接P2P传输。

| 维度 | 评估 |
|------|------|
| 跨域 | 无跨域问题，P2P直连 |
| 版权 | magnet链接传播，平台不存储内容 |
| 带宽 | **服务端几乎零带宽**，P2P分担 |
| 存储 | 服务端不存储文件 |
| 复杂度 | **较高**，WebTorrent集成+NAT穿透+浏览器兼容 |
| 延迟 | 初始缓冲慢，P2P网络建立需时间 |

**适用场景**：技术极客用户，大文件高清视频，5人以上同时观看

### 方案C：第三方网盘API授权（百度网盘/阿里云盘）

**原理**：用户OAuth授权登录网盘账号，通过网盘开放API获取文件列表和下载链接，前端直接播放。

| 维度 | 评估 |
|------|------|
| 跨域 | **需服务端代理**获取下载链接（网盘API有Referer限制） |
| 版权 | **风险较高**，用户网盘内容版权不明 |
| 带宽 | 网盘CDN承担流量，服务端只做链接转发 |
| 存储 | 不占用自有存储 |
| 复杂度 | **高**，需对接网盘OAuth+API+限速处理+链接过期刷新 |
| 延迟 | 低，网盘CDN加速 |

**具体分析**：
- **百度网盘**：开放平台API有下载限速，非会员只有100KB/s，需服务端中转加速
- **阿里云盘**：开放API较新，下载链接有2小时过期+Referer校验，需定时刷新
- **共性风险**：网盘API政策变动风险大，可能随时调整限速或关闭接口

### 方案D：直链/URL输入（推荐辅助方案）

**原理**：用户直接输入视频URL（如.m3u8/.mp4直链），前端播放器加载播放。

| 维度 | 评估 |
|------|------|
| 跨域 | 目标服务器需允许CORS，否则需服务端代理 |
| 版权 | 用户自行提供链接 |
| 带宽 | 直连目标服务器，不占自有带宽 |
| 存储 | 零占用 |
| 复杂度 | **最低**，前端HLS.js/video.js直接播放 |
| 延迟 | 取决于目标服务器 |

**适用场景**：HLS直播流、公开视频直链

### 方案E：混合方案（推荐 ⭐）

**组合**：方案D（直链输入）+ 方案A（本地文件上传）+ 方案C（网盘授权，V2迭代）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | 直链/URL输入 | 成本最低，覆盖大部分场景 |
| P1 | 本地文件上传 | 小文件私密分享，10-500MB |
| P2 | 网盘授权 | 阿里云盘优先（API更开放），作为增值功能 |
| P3 | WebTorrent | 高级功能，大文件P2P分发 |

## 三、播放同步机制

### 3.1 主从同步模式（推荐 ⭐）

房主为"主"，其余为"从"。房主操作（播放/暂停/跳转）通过WebSocket广播给所有人。

```
房主点击播放 → WebSocket广播 { type:'play', currentTime:12.3 }
所有从端收到 → 设置video.currentTime=12.3 → video.play()
```

**事件类型**：
| 事件 | 数据 | 说明 |
|------|------|------|
| `theater:play` | { currentTime } | 播放 |
| `theater:pause` | { currentTime } | 暂停 |
| `theater:seek` | { currentTime } | 跳转进度 |
| `theater:source` | { url, type } | 切换视频源 |

### 3.2 时间轴自动纠偏

- 房主每10秒广播当前播放进度 `{ type:'sync', currentTime, playing:true }`
- 从端收到后检查偏差：`|localTime - serverTime| > 1.5秒` → 自动纠正
- 纠偏策略：渐进式调整播放速率（0.9x或1.1x），而非直接跳转（避免画面突变）

### 3.3 新加入成员同步

- 新成员加入房间时，服务端返回当前房间状态：`{ source, currentTime, playing, members }`
- 前端加载视频源后自动跳转到当前进度

## 四、实时聊天集成

### 4.1 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 复用现有聊天系统 | 不用重新开发 | 演播厅消息污染私聊记录 |
| 独立弹幕/聊天室 | 沉浸感强，与视频关联 | 需额外开发 |

### 4.2 推荐方案：独立聊天室 + 弹幕

演播厅有独立的聊天频道，同时支持：

- **右侧聊天面板**：与现有ChatView聊天UI一致，复用Socket.IO消息机制
- **弹幕模式**：聊天消息以弹幕形式飘过视频画面（可选开关）
- **消息类型**：TEXT（普通聊天）、SYSTEM（加入/离开通知）、DANMAKU（弹幕）

**Socket事件**：
```
theater:chat     → 发送/接收聊天消息
theater:danmaku  → 发送/接收弹幕
```

**数据模型**：复用现有Message表，通过roomId关联到演播厅房间

## 五、房间管理

### 5.1 数据库模型

```prisma
model Theater {
  id          String   @id @default(cuid())
  name        String   @db.VarChar(64)
  description String?  @db.VarChar(200)
  coverUrl    String?  @db.VarChar(512)
  
  // 视频源
  sourceUrl   String?  @db.VarChar(1024)   // 直链URL
  sourceType  String?  @db.VarChar(20)     // URL/UPLOAD/ALIYUN
  
  // 房主
  ownerId     String   @map("owner_id")
  owner       User     @relation("TheaterOwner", ...)
  
  // 状态
  isLive      Boolean  @default(false)
  maxMembers  Int      @default(20) @map("max_members")
  password    String?  @db.VarChar(64)     // 可选房间密码
  
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  
  members     TheaterMember[]
  messages    Message[]
  
  @@map("theaters")
}

model TheaterMember {
  id        String   @id @default(cuid())
  theaterId String   @map("theater_id")
  userId    String   @map("user_id")
  role      TheaterRole @default(MEMBER)
  joinedAt  DateTime @default(now())
  
  theater   Theater  @relation(..., onDelete: Cascade)
  user      User     @relation(...)
  
  @@unique([theaterId, userId])
  @@map("theater_members")
}

enum TheaterRole {
  OWNER    // 房主
  ADMIN    // 管理员（可踢人/禁言）
  MEMBER   // 普通成员
}
```

### 5.2 房间生命周期

```
创建房间 → 设置名称/封面/密码 → 邀请好友/分享链接
→ 加载视频源 → 开始播放（自动同步）
→ 聊天/弹幕互动
→ 房主关闭/所有人离开 → 房间标记为结束
```

### 5.3 API设计

```
POST   /api/v1/theater              创建演播厅
GET    /api/v1/theater/:id           获取房间详情
POST   /api/v1/theater/:id/join      加入房间
POST   /api/v1/theater/:id/leave     离开房间
POST   /api/v1/theater/:id/invite    邀请好友
DELETE /api/v1/theater/:id            关闭房间
PATCH  /api/v1/theater/:id           更新房间设置
POST   /api/v1/theater/:id/kick      踢人（房主/管理员）
GET    /api/v1/theater/list          演播厅列表
```

## 七、技术栈扩展

## 八、推荐方案与落地步骤

### 推荐：方案E混合方案（直链优先+本地上传+网盘V2迭代）

### 落地步骤

**Phase 1（MVP，2-3天）**：
1. 数据库：Theater + TheaterMember模型
2. 后端：CRUD API + Socket.IO房间事件
3. 前端：演播厅列表页 + 创建页 + 播放房间页
4. 视频：直链URL输入 + video.js播放器
5. 同步：WebSocket主从同步（play/pause/seek/sync）
6. 聊天：复用Socket.IO消息机制，右侧聊天面板

**Phase 2（完善，3-5天）**：
7. 弹幕引擎：Canvas 2D弹幕渲染
8. 本地文件上传：tus分片上传 + 服务端流式分发
9. 邀请好友：从好友列表选择 + 房间密码
10. 成员管理：踢人/禁言/权限

**Phase 3（增值，后续迭代）**：
11. 阿里云盘授权：OAuth + API文件列表 + 代理下载链接
12. WebTorrent：P2P大文件分发
13. 直播模式：HLS推流 + 观众观看

## 九、版权与合规说明

- 演播厅仅提供技术基础设施（播放器+同步+聊天），不存储/分发任何视频内容
- 直链模式：用户自行提供URL，平台不承担内容审核义务
- 本地上传：文件由用户上传，平台提供"通知-删除"机制
- 网盘授权：用户授权访问自己的网盘文件，平台不缓存内容
- 建议在创建房间时显示免责声明，提醒用户遵守版权法规
